# Dogfood: Django 5.1.4 — L0e MRO validation (the killer demo)

**Date**: 2026-06-17
**Target**: `django/django @ tag 5.1.4` (~3 292 indexable files, 879 production .py + 1 903 tests/, 39 688 symbols, 192 576 refs)
**Binary**: `abyss 0.5.0` (slim, release build, main HEAD `f967979`)
**Hypothesis under test**: The L0e Python MRO walker, which fired **0 times on FastAPI** (external bases) and **2 times on click**, was sold as load-bearing for projects whose inheritance chains stay in-repo. Django is the textbook case — `Model` → `ModelAdmin` → `View`/`ListView`/`CreateView`, `Form` → `ModelForm`, `SimpleTestCase` → `TransactionTestCase` → `TestCase`. Predicted ≥100 L0e hits.

## TL;DR

> L0e fired **9 450 times** on Django 5.1.4. Hypothesis **violently confirmed** (94× the floor prediction). Of those, ~536 land inside `django/` proper (production–production resolutions), the rest are tests/ subclassing `django.test.TestCase` and friends — also legitimate in-repo MRO walks. FastAPI=0 / click=2 / Django=9 450 maps exactly to the "external roots vs in-repo roots" thesis from the FastAPI surprises section.

Three new findings shake out of the run, none of which kill the demo:

1. **`callers <BaseClass>` does not surface `inherit` edges.** Django has 6 838 high-confidence inheritance edges in the index, but `abyss callers Model` returns 7 (just the call/instantiation users). The vite `ViteDevServer` bug pattern recurs: a kind exists in the DB and is invisible at the agent surface. `--types-only` only matches `type_ref`, not `inherit`.
2. **L0e collides on sibling-name classes.** `DatabaseSchemaEditor` is defined in 4 backends (oracle/mysql/postgresql/sqlite3); every `self.execute()` call from any backend resolves into `postgresql/schema.py` — a precision miss when the symbol-name → file mapping has multiple candidates.
3. **Module labeller still pulls hub files into test-dominated clusters.** `django/db/models/base.py` (308 inbound refs, the actual hub) lands in `module=tests-27`. Same root cause as FastAPI's `cluster-2` swallowing `routing.py`.

**Verdict: 8 / 10.** Up from FastAPI's 6.5 because the central hypothesis landed cleanly and the data on the page is undeniable; held below 9 because the inherit-edge gap is the *exact* shape of the v0.5.0 `kind='call'` bug we already fixed once and the L0e collision case needs an L0e-disambiguation tier before the next eval pass.

## Card body — `django/db/models/base.py` (full completeness analysis)

The pre-edit card for `Model`'s definition file (the hub of the Django ORM, 308 inbound refs) is the most load-bearing artifact in the index. Full body printed above. Completeness assessment:

**What it gets right**:
- `HIGH BLAST RADIUS` flag fired on 101 prod callers — exactly the prompt the agent needs before touching this file.
- Contracts list (25 exported callables) is dedup'd correctly — `save`, `save_base`, `asave` show up once each despite being defined inside multiple inner classes.

**What it misses**:
- `Model (class) → [0 callers, 0 tests]` — this single line is the most expensively-wrong number in the report. There are **983 inheritance refs** to `Model` in the DB. The card uses the same broken `callers` path as Probe B (no `inherit` consumed).

- Card doesn't mention that `Model` is a base for ≥ 800 in-repo classes (would let the agent know "edit affects every Model subclass" instead of just "101 import users").

**Verdict**: 8 / 10 on the card axis. After B1 fixes inherit-edge surfacing, the same card would jump to ≥ 9 — the BLAST RADIUS warning text would change from "101 prod callers" to "101 callers + 983 subclasses spanning django/contrib/auth, gis, sessions, redirects, …" and the contracts list would dedup-merge `save` callers across in-repo overrides.

## Bugs / UX debts found

### B1 — `kind='inherit'` is invisible to `abyss callers` (headline bug)

**Symptom**: `abyss callers Model` returns 7 hits; the DB has 983 inherit refs at confidence ≥ 0.9 targeting `Model`. `--types-only` doesn't fix it (matches `type_ref`, not `inherit`).

**Why this matters**: For class-heavy Python (Django, SQLAlchemy, Pydantic-derived) and Java/C++ codebases, "who subclasses this?" IS the question. Inheritance is the load-bearing relation. We have the data; we don't surface it.

**Reproducer**:
```sh
abyss --workspace /tmp/django callers Model               # 7 hits
abyss --workspace /tmp/django callers Model --types-only  # 0 hits
# but in DB: SELECT COUNT(*) FROM refs WHERE kind='inherit' AND target_name='Model' AND confidence>=0.9
# → 882
```

**Fix sketch**: `callers` default should be `kind IN ('call', 'field_access', 'type_ref', 'inherit')`. Or add a new flag `--inherits-only` for the common Django case. Either way, the default behaviour needs to capture `inherit` — same lesson as v0.5.0's `kind='call'` default.

### B2 — L0e collides on sibling-name classes (precision miss)

**Symptom**: `DatabaseSchemaEditor` is defined in 4 backends (`oracle/schema.py`, `mysql/schema.py`, `postgresql/schema.py`, `sqlite3/schema.py`). Every `self.execute()` from ANY backend resolves into `postgresql/schema.py` — including 5 hits from `oracle/`, 2 from `mysql/`, 5 from `sqlite3/`.

**Why this matters**: An agent reading `oracle/schema.py` and tracing `self.execute()` lands in the wrong sibling file. The MRO walker found a `DatabaseSchemaEditor` class — but picked the wrong one because it didn't condition on "which `DatabaseSchemaEditor` does the source file see?"

**Reproducer**:
```sql
SELECT sf.path, COUNT(*) FROM refs r
JOIN files sf ON r.source_file_id = sf.id
WHERE r.receiver_type='DatabaseSchemaEditor' AND r.target_name='execute'
GROUP BY sf.path;
-- oracle/schema.py: 5  → all landed in postgresql/schema.py
-- mysql/schema.py: 2  → same
-- sqlite3/schema.py: 5 → same
-- postgresql/schema.py: 7 → correctly landed in self
```

**Fix sketch**: L0e tier needs a "prefer the `DatabaseSchemaEditor` in the same directory as the caller" disambiguation step before walking MRO. Currently it picks first-globally. The same fix pattern as L2 (same-pkg-unique).

### B3 — Module labeller pulls hub files into test-dominated clusters

**Symptom**: `where django/db/models/base.py` → `module=tests-27`. Same shape as FastAPI's `module=cluster-2` for `routing.py` — bidirectional clustering sees the hub-class has 1 800+ inheritance edges from tests/ and groups it with them.

**Fix sketch**: Either weight `inherit`/`type_ref` differently from `call` in clustering, or prefer the source-file's directory prefix as a tiebreaker when cluster membership is ambiguous. Already on the FastAPI follow-up list — Django reproduces it on a bigger corpus and confirms it's not a Python-3 special case.

### B4 — Warm-path re-resolve runs even with 0 new refs

**Symptom**: Second `abyss index` with no file changes still spends 1 470 ms in resolve (cold was 5 234 ms — so ~30 % of cold cost is re-running tier UPDATE cascade against unchanged data).

**Why this matters**: For a workspace where most files are unchanged, the resolve step should be a near-no-op. 1.5 s per re-index is fine for daemon mode but adds up for hook-triggered re-indexes.

**Fix sketch**: Guard tier cascade behind `changed_files > 0` (insert step already does this). Or partition refs by file_id and only re-resolve for changed files (cascade currently scans the whole table).

### B5 — `context` (and card) `depends-on` leaks L4 mis-resolutions

**Symptom**: `depends-on` for `views/generic/list.py` shows `→ all (django/template/utils.py) → update (django/contrib/sessions/backends/base.py) → exists (django/contrib/sessions/backends/cached_db.py) → append (django/contrib/gis/geos/mutable_list.py)`. None of these are real dependencies of `ListView` — they're bare-name dict/queryset operations the L4 global-unique tier landed on arbitrary winners.

**Fix sketch**: For `depends-on` surfaces, filter to confidence ≥ 0.8, OR exclude refs with target_qualifier=NULL when target_name is a 2-letter or super-common method name (`pop`, `items`, `update`, `append`, `get`, `set`, `all`, `exists`).

### B6 — `callers` with 0 results gives no explanation footer

**Symptom**: `abyss callers ListView` returns `no callers found for 'ListView'`. True statement (Django doesn't subclass ListView in-repo), but an agent can't tell whether that's correct ("ListView is a leaf in this repo") or wrong ("the resolver missed something").

**Fix sketch**: When `callers` returns 0, append a one-line footer: `0 high-confidence callers. DB also has: 4 inherit refs (qualifier='generic', conf 0.5), 1 import_binding. Try --min-confidence 0.5 or check qualified usage.`

## Final verdict 0 – 10

**8 / 10.**

abyss handles a 3 292-file / 39 K-symbol / 192 K-ref Django checkout in **6.9 s cold / 1.6 s warm**, hook latency 10 ms, peak RSS 217 MB — every scale target lands. The headline result is the hypothesis test: **L0e fired 9 450 times** on Django vs 0 on FastAPI, exactly because Django's class roots (`Model`, `ModelAdmin`, `View`, `Form`, `BaseDatabaseSchemaEditor`, `SessionBase`, `BaseCache`, `Storage`, `Func`, `BaseCommand`) all live in-repo.
