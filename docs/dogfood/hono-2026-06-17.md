## 5. v0.5.0 improvements visible in this dogfood

This is the first dogfood run on v0.5.0, and three changes are directly observable in the outputs above:

1. **G1 — `callers` now includes `type_ref` rows.** Probe d (`callers Context`) returns 20 results, all annotated `(100%, type)`. On v0.4.0 the exact same command on the exact same DB would have returned **zero** — this is the vite-report's headline finding ("`callers ViteDevServer` returns nothing") closed. The annotation `(100%, call)` vs `(100%, type)` lets the agent distinguish runtime vs type positions. The pre-edit card body also surfaces this split as `← 151 prod callers (46 call, 105 type)` — call/type ratio is now first-class context.

2. **G2 / G3 — test-aware ranking + L4 test-skip.** Probe e (`search "middleware"`) returns zero `*.test.ts` files in the top 10, despite hono having `compose.test.ts`, `context.test.ts`, `hono.test.ts`, etc. all matching the keyword strongly. Probe d's `(20 prod, 6 tests excluded — use --include-tests to see all)` preamble shows the test-exclusion default is wired into `callers` too. On v0.4.0, search would have surfaced 2-3 test files in the top 5, and `callers` would have mixed test references into the main list.

3. **Monorepo-aware module labelling.** The 74 clustered modules show prefixes like `cluster-N`, `unlabelled-N`, `middleware-N`, `adapter-N` — the v0.4.0 / vite-era `p-N` (truncated `packages/` prefix) and `mixed:create-vite+-0` (raw merge bookkeeping) labels are gone. The new labels still aren't human-meaningful in every case (`middleware-10` is the 10th member of the middleware cluster, where `context.ts` happens to land), but they're internally consistent and tell the user "no good label" honestly via `cluster-N` / `unlabelled-N` rather than leaking implementation detail.

Net: the three vite-report headline gaps (TS type blindness, search test pollution, monorepo labelling regression) all show measurable improvement on this hono rerun. The dogfood specifically validates that v0.5.0 closes the W1 UX debts the original hono run surfaced.

## 6. Bugs / UX debts discovered

1. **`callers` display cap is silent.** `callers Context` reports "20 prod, 6 tests excluded" but the DB has 235 high-confidence refs to `Context` (53 call + 182 type_ref, across 5 distinct `Context` symbols). The CLI silently truncates to 20. Either show `(showing 20 of 235)` in the footer or surface a `--limit N` / `--all` flag.

2. **`callers --include-tests` count is incorrect.** Without `--include-tests`: `20 prod, 6 tests excluded` (implying 26 total). With `--include-tests`: `20 found` and 20 rows printed. Either the prod count overlaps with tests (double counting) or `--include-tests` is applying the same 20-row cap on a wider universe. Both interpretations are buggy.

3. **`Set (interface)` in contracts collides with JS built-in `Set`.** The `src/context.ts` card shows `Set (interface) → [17 callers, 5 tests]`. Hono defines a `Set` interface; the resolver attributes uses of the JS global `Set` to the Hono symbol because of bare-name match at L4 / L5. Fix: when the symbol name matches a JS built-in lexicon (`Set`, `Map`, `Promise`, `Error`, `Date`, `RegExp`, `Array`), bias the resolver toward leaving unresolved rather than collapsing to the local definition.

4. **`impact direct=2` vs `callers prod=20` gap is opaque.** Same target (`Context`), same DB, same confidence threshold. `impact` reports 2 direct callers, `callers` reports 20. They're applying different definitions of "direct caller" and the user has no way to know which. Document the contract, or unify the two commands' caller selection.

5. **Layer dictionary blind to framework vocabulary.** `where src/hono.ts` and `where src/context.ts` both return `layer=unknown conf=0.00`. The current dictionary doesn't know that `hono.ts` is the framework entry or `context.ts` is the request-scoped state object. Same gap as vite's `build.ts` and helix's many `unknown` files — the layer dictionary needs framework-aware vocabulary extension or a "name matches a top-level file" fallback.

6. **`module=middleware-10` carries no meaning.** Better than v0.4.0's `p-10` (no truncation), but still labels `src/context.ts` by which cluster it landed in rather than what it is. `cluster-0..15` and `unlabelled-23` are now also visible — those are honest about "we couldn't name this", which is the right behaviour. The `<word>-N` form is the ambiguous middle case.

7. **`context` command duplicates same-name imports.** `bearerAuth`'s `depends on:` list shows `→ Response (src/client/types.ts)` twice — two distinct positional refs collapsing to the same display row. Same root cause as the vite `waitForRequestsIdle` duplication. Group identical (target_name, target_file) pairs before display.

8. **High unresolved fraction on TSX (66.8%).** TypeScript unresolved is 58.9% (in line with vite's 67.9%), but TSX is 66.8% — the JSX expression resolver is leaving more on the table. Likely because JSX intrinsic elements + component refs need their own L0-tier binding pass.

None of these are correctness regressions. The bugs are coverage/display issues (1, 2, 4, 7), labelling debt (5, 6), one resolver false-positive class (3), and one unresolved-rate observation (8).

## 7. Final verdict: **8 / 10**

abyss indexed a 388-file TypeScript-first framework in **793 ms cold, 333 ms warm**, zero crashes, zero degraded-mode warnings, p95 probe latency under 25 ms — the structural pipeline is solid. The two probes that drove the most value were (1) `callers Context` which is now non-zero thanks to v0.5.0 G1's `type_ref` inclusion (this was the vite-report's headline gap, and it's closed — 20 prod callers all correctly annotated `(100%, type)`), and (2) the `src/context.ts` pre-edit card which delivers `151 prod callers (46 call, 105 type) across 43 files` in 23 ms — a complete impact map an agent could never reconstruct from grep. Search ranking is clean (zero test pollution in top 10 for "middleware", direct G2/G3 wins), and the cross-language family filter holds (1,974 jsfam→jsfam edges, zero cross-family leaks). The remaining gaps are presentation debt: the `callers` command silently caps at 20 rows even when 235 are available, `impact` and `callers` disagree on what "direct caller" means, the `Set` interface collides with the JS built-in, and `module=middleware-10` still isn't a human label. These don't block agent use — an LLM editing hono today gets real, actionable signal from every probe — but a `--limit` flag, a built-in name-shadow filter, and one more pass on the module labeller would push this from 8 to 9. The dogfood validates that v0.5.0 specifically closed the W1 UX debts this codebase originally surfaced.
